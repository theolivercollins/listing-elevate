## 1. Goal

Add **Ally** — Listing Elevate's AI concierge — to the public homepage (`/` → `<V2Landing />` at `https://www.listingelevate.com`). She offers help, answers product questions, qualifies interested visitors, and routes high-intent leads to sign-up. She is the same brand "Ally" as the in-app blog assistant — same name, similar visual language — but a different surface, audience, and job.

**Not goals (v1):**
- Live human takeover / live chat staffing.
- "Show me what you'd do with my listing" demo (paste MLS link → sample treatment). Big build, deferred.
- Calendly "book a call" CTA. Wire-up deferred until a Calendly URL exists; only `get_started` CTA in v1.
- Voice / phone / SMS surfaces.
- Multi-language.

---

## 2. Hard rules (do not violate)

1. **No Helgemo Team branding.** Listing Elevate is multi-tenant SaaS; Helgemo Team is a customer (and the founder's team), not the brand. Strip all `Helgemo Team`, `Punta Gorda`, `Charlotte County`, `Burnt Store Isles`, `The Isles`, `we manage every part of...` Helgemo voice from the lifted system prompt. Ally on this surface is a generic "Listing Elevate concierge for real estate agents."
2. **Never fabricate.** Pricing, turnaround, feature claims, founder story — all sourced from the curated knowledge file + per-section JSON fragments. No hallucinated stats.
3. **Never pretend to be human.** If asked, Ally says: *"I'm Listing Elevate's AI concierge — but a real human's an email away if you'd rather."*
4. **Anonymous-friendly.** No login, no email required to talk. Soft email request after turn 3 only.
5. **Cost-tracked.** Every turn writes to `cost_events` with `stage='marketing_chat'`, `provider='anthropic'`. Per-session $1.00 hard cap enforced server-side.
6. **Single accent color.** LE design uses one accent (electric blue, HSL `220 90% 56%`). Do not introduce a second accent for Ally.

## 4. Persona, system prompt, response schema

### 4.1 Persona

Ally is **Listing Elevate's concierge**. Warm, knowledgeable, low-pressure. She knows the product cold, answers honestly, and gently routes interested visitors toward sign-up. She speaks for "Listing Elevate" or "we" (the team behind LE), never as an individual person. She never name-drops Helgemo Team or any specific customer.

### 4.2 Voice rules

- Warm + low-pressure. No exclamation-mark spam. No "Absolutely!" / "Great question!" filler.
- One question at a time.
- Brief — 1–4 sentences per reply unless walking through pricing or process.
- Volunteers a CTA when intent is clear, never on every turn.
- Uses **only** facts present in the knowledge base + pricing + FAQ. If asked something not in the knowledge, says so plainly: *"I don't have that locked down — want me to flag it for the team to follow up over email?"*
- If asked who/what she is: *"I'm Listing Elevate's AI concierge — happy to answer most things, but a real human's one email away if you'd rather."*

### 4.3 System prompt structure (assembled per request)

```
[BASE_PROMPT]                    ~600 tokens, cached
  - persona, voice rules, schema spec, refusal rules

[KNOWLEDGE]                      ~800 tokens, cached
  - lib/marketing/knowledge.md verbatim

[PRICING]                        ~200 tokens, cached
  - JSON.stringify(pricing.json)

[FAQ]                            ~400 tokens, cached
  - JSON.stringify(faq.json)

[CONVERSATION_META]              ~50 tokens, NOT cached
  - turn_count, has_email, source_url, current_section_hint
```

Total system prompt: ~2KB / ~2000 tokens. With Anthropic prompt caching enabled (`cache_control: { type: 'ephemeral' }` on the cached blocks), the static portion is ~$0.0001/turn instead of ~$0.012/turn.

### 4.4 Response schema (parsed server-side)

```
<reply>
1–4 sentences of plain prose. Always present.
</reply>

<ally_followup_chips>
Up to 3 short suggestions, semicolon-separated. e.g.
"How long does turnaround take?; What do you need from me?; Show me pricing"
Emit when the conversation has natural next questions; omit on closing turns.
</ally_followup_chips>

<ally_cta>
One word: get_started
Emit when the visitor has clearly signaled intent — "how do I sign up", "what's next",
"can I try it", "I want one for my listing". Omit on info-gathering turns.
When emitted, ALSO mention it in <reply>.
(book_call deferred until a Calendly URL is wired.)
</ally_cta>

<ally_lead_capture>
JSON: {"name": "...", "email": "...", "phone": "...", "role": "agent|broker|other", "intent": "..."}
Emit ONLY when the visitor has volunteered fields in conversation. Never invent values.
Never include a field the visitor didn't actually share.
</ally_lead_capture>
```

### 4.5 Identity gate (soft, turn-3)

- Turns 1–3: Ally answers freely, never asks for contact info.
- Turn 4+: if no email captured AND visitor has asked ≥2 product-substantive questions, Ally adds one soft line at the end of her reply: *"By the way — want me to email you a one-pager you can come back to?"*
- Never blocks the conversation. Never repeats the soft-ask more than twice in a session.
- Implementation: `conversation_meta.turn_count` and `conversation_meta.has_email` are passed in the system prompt; the soft-ask is enforced by prompt instruction, not by a hard server-side gate.

## 6. Data model

### 6.1 `marketing_leads`

```sql
create table marketing_leads (
  id uuid primary key default gen_random_uuid(),
  conversation_id uuid not null unique,   -- matches the cookie
  email text,
  name text,
  phone text,
  role text,                              -- 'agent' | 'broker' | 'other' | null
  intent text,                            -- 1-line summary Ally extracted
  conversation jsonb not null,            -- full message thread, capped at 30 messages
  source_url text,
  ip_hash text,                           -- sha256(ip + IP_HASH_SALT) — never raw IP
  user_agent text,
  utm jsonb,                              -- {source, medium, campaign, term, content}
  total_messages int not null default 0,
  total_cost_cents int not null default 0,
  status text not null default 'new',     -- 'new' | 'contacted' | 'converted' | 'spam'
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index marketing_leads_email_idx on marketing_leads (email) where email is not null;
create index marketing_leads_created_idx on marketing_leads (created_at desc);
create index marketing_leads_status_idx on marketing_leads (status);

alter table marketing_leads enable row level security;
-- No policies: only the API route's service-role client can read/write.
```

### 6.2 `marketing_chat_rate_limits`

```sql
create table marketing_chat_rate_limits (
  bucket_key text primary key,            -- e.g. "ip:abc123:minute:2026051502"
  count int not null default 0,
  window_start timestamptz not null default now(),
  expires_at timestamptz not null
);

create index marketing_chat_rate_limits_expires_idx on marketing_chat_rate_limits (expires_at);

-- Janitor: a daily Vercel cron (or Supabase pg_cron) deletes rows where expires_at < now().
```

### 6.3 Cost ledger

Reuse existing `cost_events` table. Verified columns (from `lib/blog-engine/cost.ts` + migration `048_blog_engine.sql`): `stage text`, `cost_cents int`, `post_id uuid (nullable)`, `site_id uuid (nullable)`, `provider text (CHECK includes 'anthropic')`, `metadata jsonb`. **No schema change needed** — `stage` is unconstrained text, `site_id`/`post_id` are nullable.

```sql
insert into cost_events (stage, provider, cost_cents, post_id, site_id, metadata)
values (
  'marketing_chat',
  'anthropic',
  <cost_cents>,
  null,
  null,
  jsonb_build_object(
    'conversation_id', '<conversation_id>',
    'model', 'claude-sonnet-4-6',
    'input_tokens', <input_tokens>,
    'output_tokens', <output_tokens>,
    'ip_hash', '<ip_hash>',
    'source_url', '<url>',
    'cache_hit', <bool>
  )
);
```

### 6.4 Conversation persistence

Single table — `marketing_leads` is also the transient thread store. No second table.

- On first turn: server generates `conversation_id` (uuid) and sets `mally_cid` cookie (HttpOnly, SameSite=Lax, 30d, Secure in prod). Server INSERTs a `marketing_leads` row with `email=null`, `name=null`, `status='new'`, full thread snapshot in `conversation`.
- Each subsequent turn: server UPDATEs the same row, appending to `conversation`, bumping `total_messages`, `total_cost_cents`, `updated_at`.
- On `<ally_lead_capture>`: same UPDATE, plus patches the volunteered fields (email/name/phone/role/intent).
- TTL: handled by the shared `/api/cron/marketing-ally-cleanup` cron (see §7) — deletes rows where `updated_at < now() - interval '90 days' AND email IS NULL`.

## 7. Rate limiting (Supabase, no new infra)

```
Per IP (ip_hash):
  5 requests / minute       (burst limit — abuse signal; exceeds → 429 with Retry-After)
  50 sessions / day         (one IP = one office; exceeds → 429, gentle "we've helped a lot of folks from your network today")

Per conversation_id:
  30 messages / session     (matches blog Ally cap; exceeds → Ally closes with "let's pick this up over email")
  $1.00 / session hard cap  (server tracks cumulative cost_cents; exceeds → same close-out)

Global (kill switch):
  500 conversations / day   (env var MARKETING_ALLY_DAILY_CAP; exceeds → endpoint returns 503 + alerts you)
```

**Implementation** — `lib/marketing/rate-limit.ts`:
- One `assertRateLimit(ipHash, conversationId)` function called at the top of the handler.
- Each bucket is a row in `marketing_chat_rate_limits` keyed `<scope>:<id>:<window>`.
- Window keys: minute = `YYYYMMDDHHMM`, day = `YYYYMMDD`.
- Atomic increment via Postgres `INSERT ... ON CONFLICT (bucket_key) DO UPDATE SET count = count + 1 RETURNING count`.
- If returned `count` exceeds limit → throw `RateLimitError` → handler returns 429.

**Janitor:** A single daily Vercel cron at `/api/cron/marketing-ally-cleanup` (registered in `vercel.json`) does two passes:
1. `delete from marketing_chat_rate_limits where expires_at < now()` — purges expired rate-limit buckets.
2. `delete from marketing_leads where updated_at < now() - interval '90 days' and email is null` — purges abandoned anonymous threads but keeps captured leads forever.

~1 second/day total. Idempotent; safe to retry.

## 8. Knowledge build pipeline

### 8.1 `lib/marketing/knowledge.md` (curated, hand-written)

~2KB markdown, sections:
- **What is Listing Elevate?** — one paragraph product summary.
- **What you get** — bullet list of deliverables (cinematic mp4, 9:16 / 16:9, X scenes, etc.).
- **Process** — 3-step description (upload → AI generates → delivered via email).
- **Turnaround** — current SLA in plain English.
- **What we need from you** — photo requirements, listing details, optional voiceover script.
- **Who is Listing Elevate for?** — listing agents, teams, brokerages, photographers.
- **Founder story** — 2 sentences, brand-neutral (not Helgemo-specific).
- **What Ally cannot do** — explicit list so she refuses gracefully (e.g., "I can't book a video for you yet — sign up and you'll be set up in 60 seconds").

Versioned in git. Update it when the product changes; Ally is automatically smarter on next deploy.

### 8.2 `lib/marketing/build-knowledge.ts` (build-time)

- Implementer first refactors `Pricing.tsx` and `FAQ.tsx` to export named constants: `export const PRICING_TIERS = [{ name, price, includes, ... }]` and `export const FAQ_ITEMS = [{ q, a }]`. The component continues to render from these constants (no behavior change).
- Build script `tsx`-imports the two files and `JSON.stringify`s the constants into `lib/marketing/pricing.json` and `lib/marketing/faq.json`.
- Runs in the `build` npm script, before `vite build`. If the named exports are missing, build fails loudly — no silent stale knowledge.

### 8.3 Why both (curated + extracted)

- Curated knowledge.md: things that don't live in any single component (founder story, "what we cannot do", value prop).
- Extracted JSON: things that have a canonical source on the page (pricing tiers, FAQ items) and **must not drift** from what the visitor sees.

## 9. Cost projection

Per-turn (Sonnet 4.6, prompt caching ON):

| Component | Tokens | Cost (cached) |
|---|---|---|
| System prompt (cached after turn 1) | ~2000 in | ~$0.0002 |
| Per-turn meta + user message | ~200 in | ~$0.0006 |
| Output | ~300 out | ~$0.0045 |
| **Total per turn** | | **~$0.005** |

- 30-msg session: ~$0.15
- 100 sessions/day: ~$15/mo
- 500 sessions/day (kill-switch ceiling): ~$75/mo

All recorded in `cost_events`, visible in your existing dashboard and reconcilable against the Anthropic invoice.
