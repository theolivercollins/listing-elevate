# v1.1 — Seedance push-in pipeline + FFmpeg speed-ramp polish

**Date:** 2026-05-23
**Owner:** Oliver
**Branch:** `feat/v1-1-seedance-pushin` (off `dev`)
**Status:** Draft for approval
**Touches v2.1?** No. Paired-frame rule (router.ts:242 → `atlas + kling-v2-1-pair`) is preserved verbatim.

## Goal

Add an opt-in **pipeline mode `v1.1`** that:

1. Routes every **non-paired** clip to **Seedance** (Bytedance) image-to-video.
2. Forces the camera movement to `push_in` regardless of what the prompt-grading or scene script chose.
3. Applies a subtle **FFmpeg speed-ramp polish** to the head and tail of every clip after download, before upload to storage.

Toggleable per-property from the order form. Default = `v1` (existing behavior). Production-only; not exposed in the Lab.

## Non-goals (YAGNI)

- Seedance for **paired** scenes. Paired scenes always go to Kling 2.1 (existing rule).
- Movement variety under v1.1. It's push_in only, full stop. No orbit / parallax / etc.
- Per-scene model overrides. The toggle is per-property.
- Lab Picker exposure. Lab continues to use the existing V1 Atlas SKUs.
- Thompson router changes. v1.1 short-circuits before Thompson.
- Backfilling speed-ramp onto already-uploaded clips. Re-renders only.

## Architecture

### 1. Provider — `lib/providers/seedance.ts` (NEW)

New `SeedanceProvider implements IVideoProvider`. Hosted via **Replicate** (`bytedance/seedance-1-pro`). Replicate is the cheapest stable host for Seedance and `REPLICATE_API_TOKEN` is already a slot in `docs/CREDENTIALS.md`.

- `name: "seedance"` — added to the `VideoProvider` union in `lib/types.ts`.
- `generateClip()` — POSTs to `https://api.replicate.com/v1/predictions` with `version: <pinned-seedance-1-pro-version-sha>`, `input: { image: sourceImageUrl, prompt, duration: durationSeconds, resolution: "1080p", aspect_ratio: "16:9" }`. Ignores `endImageUrl` (Seedance doesn't support it). The provider prompt is overridden by the caller to "slow push in" regardless of the scene's text — see `forceSeedancePushInPrompt()` in `lib/providers/router.ts`.
- `checkStatus()` — GET on Replicate prediction. `status: 'starting'|'processing'` → `processing`; `'succeeded'` → `complete` with the `output` URL; `'failed'|'canceled'` → `failed`.
- `downloadClip()` — straight HTTP GET.
- Cost: Replicate bills per second of generated video. Initial constant: `SEEDANCE_CENTS_PER_SECOND = 12` (matches their published rate, $0.12/s for `seedance-1-pro` 1080p as of 2026-05; verify on first invoice). Tracked via `cost_events` like every other provider.

### 2. Router — `lib/providers/router.ts` (EDIT)

A new top-level branch in `selectProviderForScene()`, **above** the paired rule, **only** when `mode === 'v1.1'`:

```ts
if (mode === 'v1.1' && !scene.endPhotoId) {
  return { provider: 'seedance', modelKey: 'seedance-1-pro-pushin', fallback: { provider: 'atlas', modelKey: V1_DEFAULT_SKU, fallback: undefined } };
}
```

Paired scenes (`scene.endPhotoId` truthy) **fall through to the existing rule** even under `v1.1` — paired always wins, v2.1 untouched.

`selectProviderForScene` gains an optional `mode: 'v1' | 'v1.1'` parameter, defaulted to `'v1'`. All existing call sites continue to work unchanged.

Fallback to Atlas `kling-v2-6-pro` on Seedance permanent failure so a v1.1 property doesn't dead-end on a provider outage.

`getEnabledProviders()` gains `if (process.env.REPLICATE_API_TOKEN) enabled.push('seedance')`.

### 3. Forced push-in prompt — `lib/providers/router.ts` (NEW EXPORT)

```ts
export function forceSeedancePushInPrompt(originalPrompt: string): string {
  // Strip any curve/orbit/parallax language; replace with a single, stable
  // push-in directive. Keeps the scene's subject description.
  return `Slow, steady push in toward the room. Camera moves smoothly forward on a fixed dolly. No tilt, no rotation, no parallax, no orbit. ${stripMovementVerbs(originalPrompt)}`;
}
```

`stripMovementVerbs` is a small regex over a closed list (orbit, rotate, tilt, pan, parallax, swing, sweep, dolly out, pull back, fly through, etc.). Returns the prompt with those verbs and their adjacent adverbs removed.

### 5. Pipeline plumbing — `lib/pipeline.ts` (EDIT)

- Submit loop loads `properties.pipeline_mode` once per property (already part of the `property` row fetch — just add the column to the select list).
- Passes `mode` into `selectProviderForScene({ ..., mode })`.
- When provider returned is `seedance`, overrides the `prompt` arg with `forceSeedancePushInPrompt(scene.prompt)` before calling `generateClip()`. The scene's stored prompt is **not** mutated — the override is render-time only, so the prompt audit trail in `scenes.prompt` remains the human-authored one.

### 6. Speed-ramp polish — `lib/utils/ffmpeg.ts` (EDIT) + cron `api/cron/poll-scenes.ts` (EDIT)

New function:

```ts
export async function applySpeedRamp(
  inputPath: string,
  outputPath: string,
  opts?: { rampSeconds?: number; rampFactor?: number },
): Promise<void>;
```

Default: `rampSeconds = 0.5`, `rampFactor = 0.8` (final 20% slowdown at the very ends).

Implementation: three `trim` segments concatenated with per-segment `setpts`:

```
[0:v]trim=0:0.5,setpts=PTS/0.8+STARTPTS[head];
[0:v]trim=0.5:DUR-0.5,setpts=PTS-STARTPTS[mid];
[0:v]trim=DUR-0.5:DUR,setpts=PTS/0.8+STARTPTS[tail];
[head][mid][tail]concat=n=3:v=1[out]
```

Wait — `setpts=PTS/0.8` *speeds up*. For a slow ease, we want `setpts=PTS/0.8` → 1.25× duration → 0.8× speed. Confirmed: dividing PTS by a value < 1 stretches the clip (slows it). So `/0.8` = 80% speed = subtle slow-in / slow-out. (Test in `applySpeedRamp.test.ts` verifies the output duration is longer than input by `2 * rampSeconds * (1/rampFactor - 1)`.)

Cron change in `api/cron/poll-scenes.ts` around line 114:

```ts
const clipBuffer = await provider.downloadClip(status.videoUrl);
const finalBuffer = (property.pipeline_mode === 'v1.1')
  ? await applySpeedRampToBuffer(clipBuffer)
  : clipBuffer;
// ... upload finalBuffer instead of clipBuffer
```

`applySpeedRampToBuffer` is a thin wrapper that writes the buffer to a `tmp` file, calls `applySpeedRamp`, reads the result back, and cleans up. Implemented in `lib/utils/ffmpeg.ts`.

The original (un-ramped) clip is **not** persisted. v1.1 ramp is part of the clip itself. (Rationale: keeps storage costs flat and pipeline simple. If we ever want A/B against raw clips we'll add a `clip_raw_url` column then.)
