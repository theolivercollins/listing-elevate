# v1.1 — Quality toggle + Veo 3.1 Premium 4K + qualitative model feedback

**Date:** 2026-05-24
**Branch:** `feat/v1.1-quality-veo-feedback` (off `main`)
**Status:** Draft for approval
**Predecessors:** `2026-05-24-prompt-lab-v1.1-director-design.md`, `2026-05-23-prompt-lab-version-toggle-design.md`

## Goal

Three additions to v1.1 sessions:

1. **Per-model quality dropdown.** After picking a SKU, the user picks a resolution from that SKU's supported set (480p / 720p / 1080p / 4K). Default per SKU is the highest the model produces well.
2. **Veo 3.1 Preview as a "Premium 4K" SKU.** New provider, Gemini API auth (same key we use for the photo analyzer), native 4K, gated behind clearer cost messaging since Veo is ~5–10× a Kling render.
3. **Qualitative feedback under each rendered clip.** Text area below the video. Saved verbatim, embedded for similarity retrieval, version-scoped, surfaced in the director's context for future renders on the same model.

## Non-goals (YAGNI)

- LLM-parsed "model insights" dashboard. Save the raw text + embedding now; do summarization later if it actually helps.
- Vertex AI auth for Veo. Use the simpler Gemini API path; Vertex would unlock GA Veo tiers and IAM-style auth but adds new cred surface.
- 4K assembly. Director composer remains 1080p concat (with FFmpeg speed-ramp). 4K source clips will downscale during assembly to keep concat compatible. If user wants 4K final output we'll add it later as a separate "final res" picker on the Director.
- Per-iteration feedback rewriting. Once saved, feedback is immutable. Delete + re-add if you want to revise.

## Architecture

### 1. Per-model quality dropdown

#### Data model

`AtlasModelDescriptor` gains:

```ts
interface AtlasModelDescriptor {
  // ... existing fields ...
  /** Resolutions the underlying model can actually produce. Order matters —
   * first entry is the default selection in the UI. */
  supportedResolutions?: ReadonlyArray<"480p" | "720p" | "1080p" | "4k">;
}
```

Each descriptor declares what it actually supports:

| SKU | supportedResolutions |
|---|---|
| `seedance-pro-pushin` | `['1080p', '720p', '480p']` |
| `kling-v3-pro` / `kling-v2-6-pro` / `kling-v2-master` / `kling-o3-pro` | `['1080p']` (Kling output res is fixed in-model) |
| `kling-v2-1-pair` | `['1080p']` |
| `runway-gen4-native` | `['720p']` |
| `veo-3-1-preview` (NEW) | `['4k', '1080p', '720p']` |

When `supportedResolutions` is unset/single, the UI hides the picker (one choice, no decision to make).

#### Render path

The render endpoint accepts `resolution` on the request body alongside `sku`. Both flow through `submitLabRender` → provider's `generateClip` → Atlas/Veo body. The current hardcode of `resolution: "1080p"` on Seedance descriptor becomes the default-when-unset; explicit per-render value overrides.

#### DB tracking

New column `resolution_used TEXT` on:
- `prompt_lab_iterations` (Sessions Lab)
- `prompt_lab_listing_scene_iterations` (Listings Lab)

Stamped at submit time. NULL on rows predating this migration — read-side code defaults to `'1080p'` for display.

#### UI

`IterationCard` gets a second select right next to the SKU select. Hidden when the current SKU's `supportedResolutions` has length ≤ 1. Switching SKU resets resolution to the new SKU's first supported value.

### 2. Veo 3.1 Preview provider

#### New `lib/providers/veo.ts`

`VeoProvider implements IVideoProvider`. Uses `GEMINI_API_KEY` (already in `.env`).

- Endpoint: `https://generativelanguage.googleapis.com/v1beta/models/veo-3.1-generate-preview:predictLongRunning`
- `generateClip()`:
  - POST `{ instances: [{ prompt, image: { bytesBase64Encoded: <b64> } }], parameters: { resolution, durationSeconds, aspectRatio } }`
  - Returns operation name as `jobId`. Veo 3.1 max duration is 8s — clamp to that.
- `checkStatus(operationName)`:
  - GET `https://generativelanguage.googleapis.com/v1beta/<operationName>`
  - On `done: true` with `response.video.uri` → `complete` + videoUrl.
  - On `done: true` with `error` → `failed`.
  - Otherwise `processing`.
- `downloadClip()`: GET the video URI. Gemini Files API returns either a signed URL or requires the API key header on the GET — handle both.
- Cost: published Veo pricing TBD against first invoice. Placeholder: `priceCentsPerSecond = 50` (covers the high end of the $0.30–$0.60/sec range Gemini's grounded research surfaced).

#### Registration

- Add `'veo-3-1-preview'` to `V1_1_LAB_SKUS` (both copies: `src/lib/labModels.ts` and `lib/providers/atlas.ts`).
- `LAB_MODELS` entry: label `'Veo 3.1 Preview (4K)'`, shortLabel `'Veo 3.1 4K'`, priceCents `250` (50¢ × 5s — note: 4K renders are ~$0.25 / 1s, $2.50 for a 5s clip; ~5× more than Seedance 1080p).
- `buildProviderFromDecision` adds a case `'veo'` mapped to `new VeoProvider()`. Veo doesn't go through Atlas — it's its own `IVideoProvider`.
- `getEnabledProviders()` adds `'veo'` if `GEMINI_API_KEY` present.

#### Render override

When `sku === 'veo-3-1-preview'`:
- Skip the Seedance push-in prompt wrap (Veo has its own prompt style; let the director prompt through unchanged).
- Skip the Lab cron speed-ramp on download (4K clips, expensive to re-encode; let assembly apply ramp at concat time only).

### 3. Qualitative model feedback

#### Schema (migration 070)

```sql
CREATE TABLE prompt_lab_model_feedback (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  iteration_id UUID NOT NULL REFERENCES prompt_lab_iterations(id) ON DELETE CASCADE,
  session_id UUID NOT NULL REFERENCES prompt_lab_sessions(id) ON DELETE CASCADE,
  model_used TEXT NOT NULL,
  pipeline_version TEXT NOT NULL DEFAULT 'v1' CHECK (pipeline_version IN ('v1','v1.1')),
  resolution_used TEXT,
  author TEXT NOT NULL,         -- supabase user id of whoever wrote it
  comment TEXT NOT NULL,        -- raw user text, no length cap
  embedding vector(1536),       -- text embedding for retrieval; backfilled async
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_plmf_model ON prompt_lab_model_feedback (model_used, pipeline_version);
CREATE INDEX idx_plmf_session ON prompt_lab_model_feedback (session_id);
CREATE INDEX idx_plmf_embedding ON prompt_lab_model_feedback USING ivfflat (embedding vector_cosine_ops);
```

One iteration can have many feedback rows (allows revising via add-new rather than edit-in-place).

#### Embedding

After insert, fire-and-forget an embed call via the existing `embedTextSafe` helper in `lib/embeddings.ts`. Backfill column when ready; null embeddings are skipped at retrieval time.

#### UI

`IterationCard` gets a collapsible "Feedback on this model" section below the video player. Shows:
- A list of prior feedback rows on THIS iteration (date + author + text).
- A `<textarea>` + Submit button. On submit, POST to new endpoint `/api/admin/prompt-lab/model-feedback`.

Optional later: aggregate view per-SKU (read-only panel showing recent feedback on `kling-v2-master` across all sessions for the current version).

#### Endpoints

- `POST /api/admin/prompt-lab/model-feedback` — body `{ iteration_id, comment }`. Server fills `session_id`, `model_used`, `pipeline_version`, `resolution_used`, `author` from the iteration row + auth. Returns the inserted row.
- `GET /api/admin/prompt-lab/model-feedback?iteration_id=<>` — returns feedback rows for one iteration.
- `GET /api/admin/prompt-lab/model-feedback?model=<>&pipeline_version=<>&limit=20` — returns recent feedback for a model (used in retrieval surfaces).

#### Retrieval wiring

New helper in `lib/prompt-lab.ts`:

```ts
export async function retrieveRecentModelFeedback(
  modelUsed: string,
  opts: { pipelineVersion: string; limit?: number; embedding?: number[] }
): Promise<Array<{ comment: string; created_at: string }>>
```

If `embedding` provided → vector-similarity search filtered by `(model_used, pipeline_version)`. Otherwise → most-recent N. Limit defaults to 5.

Wire into `per-photo-retrieval.ts` so the production director prompt includes a "Recent operator feedback on this model" block when SKU is known. Wire into `directListingScenes` similarly.

This is the "always access/reference when we make decisions" surface — the director sees the operator's qualitative notes alongside the structured rating signal.
